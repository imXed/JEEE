\section{Introduction}
Ce rapport présente une application web JEE de gestion d'un club d'escalade réalisée avec Spring Boot 3.3.5 (Java 17). Le projet répond à un cahier des charges structuré en lots : modèle de données, jeu de données de démonstration, consultation publique, espace membre sécurisé et procédure de réinitialisation de mot de passe.

L'objectif est de proposer une plateforme simple et fiable pour consulter les sorties, tout en offrant aux membres des fonctionnalités de création et de gestion. Le projet met l'accent sur la séparation des responsabilités, la validation des données et la sécurité des accès.

\newpage

\section{Architecture générale}
L'application suit une architecture en couches :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item \textbf{Présentation} : vues JSP/JSTL, navigation et formulaires.
  \item \textbf{Contrôleurs} : endpoints Spring MVC, gestion des requêtes et redirections.
  \item \textbf{Services} : logique métier, règles de validation, orchestration.
  \item \textbf{Persistance} : Spring Data JPA, entités JPA, repositories.
  \item \textbf{Sécurité} : Spring Security (form login, rôles, session).
\end{itemize}

La configuration des vues est assurée par \texttt{spring.mvc.view.prefix=/WEB-INF/jsp/} et \texttt{spring.mvc.view.suffix=.jsp}. La base utilisée est H2 en mémoire, avec génération automatique du schéma (\texttt{ddl-auto=create-drop}).

\section{Modèle de données}
L'application repose sur quatre entités principales :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item \textbf{Membre} : nom, prénom, email unique, mot de passe chiffré, rôles.
  \item \textbf{Sortie} : nom, description, site web, date, créateur, catégorie.
  \item \textbf{Categorie} : libellé de la catégorie.
  \item \textbf{PasswordResetToken} : jeton unique, expiration, état d'usage, membre.
\end{itemize}

Les relations sont gérées par JPA (\texttt{@ManyToOne}, \texttt{@OneToMany}) et la validation se fait via Bean Validation (\texttt{@NotBlank}, \texttt{@Size}, \texttt{@Email}).

\section{Fonctionnalités principales (visiteur)}
Un visiteur non authentifié peut :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item consulter la liste des catégories ;
  \item visualiser les sorties avec pagination ;
  \item accéder au détail d'une sortie (sans informations sensibles) ;
  \item filtrer les sorties via la recherche multicritère.
\end{itemize}

Les pages publiques sont accessibles sans authentification (\texttt{/}, \texttt{/sorties/**}, \texttt{/login}).

\section{Fonctionnalités principales (membre authentifié)}
Après authentification, un membre peut :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item créer, modifier et supprimer \textbf{ses propres sorties} ;
  \item consulter l'ensemble des informations d'une sortie (site web, créateur) ;
  \item accéder aux listes de membres et aux sorties par membre.
\end{itemize}

Les formulaires utilisent la validation serveur pour garantir la cohérence des données. Les opérations de modification et suppression sont vérifiées côté service afin de s'assurer que l'utilisateur est bien le créateur de la sortie.

\section{Recherche multicritère}
La recherche repose sur un objet de critères \texttt{SortieRechercheCriteria} contenant :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item nom (recherche partielle, normalisée) ;
  \item catégorie (\texttt{categorieId}) ;
  \item créateur (\texttt{createurId}) ;
  \item période (\texttt{dateDebut} / \texttt{dateFin}).
\end{itemize}

La requête est implémentée dans \texttt{SortieRepository.rechercher} via une requête JPQL dynamique avec paramètres optionnels. Une validation empêche les périodes incohérentes (date de début après date de fin).

\section{Sécurité et authentification}
La sécurité est configurée dans \texttt{SecurityConfig} :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item formulaire de connexion personnalisé (\texttt{/login}) ;
  \item identifiant = email, mot de passe = \texttt{password} ;
  \item sessions uniques avec migration pour éviter la fixation de session ;
  \item pages publiques et routes protégées clairement séparées.
\end{itemize}

Le service \texttt{CustomUserDetailsService} charge un membre à partir de son email et mappe les rôles vers des \texttt{SimpleGrantedAuthority}. Les mots de passe sont chiffrés via \texttt{BCryptPasswordEncoder}.

\newpage

\section{Initialisation des données et configuration}
Au démarrage, \texttt{DataInitializer} crée :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item 10 catégories ;
  \item 50 membres avec emails uniques ;
  \item 1000 sorties réparties sur l'année.
\end{itemize}

Les paramètres techniques sont définis dans \texttt{application.properties} : configuration JSP, génération du schéma, logs SQL, SMTP local (\texttt{localhost:10025}) et validité des tokens de réinitialisation (60 minutes).

Le format des emails est généré automatiquement à partir du prénom et du nom, en supprimant les accents, puis en ajoutant un suffixe numérique pour éviter les doublons.

\section{Réinitialisation du mot de passe}
Le flux de réinitialisation est implémenté par \texttt{PasswordResetServiceImpl} :
\begin{itemize}[leftmargin=*, label=\textbullet]
  \item génération d'un token unique ;
  \item invalidation des anciens tokens ;
  \item envoi d'un email contenant le lien de réinitialisation ;
  \item contrôle d'expiration et usage unique.
\end{itemize}
